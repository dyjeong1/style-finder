## 1. 배경/문제 정의
- 비즈니스 맥락: 개인용 로컬 서비스에서 추천 페이지가 업로드 이후 자연스럽게 이어져야 한다.
- 현재 성능/운영 이슈: 백엔드 재시작 후 메모리 업로드 기록이 사라지면, 프론트 localStorage에 남은 `uploaded_image_id`로 추천 조회가 실패한다.

## 2. 목표/가설
- 1차 지표(Primary): 오래된 업로드 ID로 추천 조회 시 치명적 오류 대신 복구 가능한 안내 메시지를 노출한다.
- 2차 지표(Secondary): 사용자가 `/upload`로 쉽게 돌아가 다시 업로드할 수 있다.
- 가설: 프론트에서 stale 업로드 상태를 감지해 자동 초기화하면 추천 흐름이 더 안정적으로 유지된다.

## 3. 범위/산출물(Scope & Deliverables)
- 포함 범위: 추천 페이지 stale 업로드 상태 감지/초기화, 문서 업데이트, 로컬 검증.
- 제외 범위: 업로드 영속 저장 구조 개편.
- 산출물(코드/모델/대시보드/문서): `frontend/app/(main)/recommendations/page.tsx`, 루트/프론트 README, TODO, PLAN/TASK 문서.
